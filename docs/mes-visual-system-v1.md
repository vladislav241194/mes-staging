# MES Visual System v1

Документ фиксирует первый общий слой визуального языка MES. Его задача - не заменить бизнес-логику и не изменить данные, а сделать интерфейс предсказуемым при развитии модулей.

## Что реально добавлено в v1

- Единая система сигналов `ready/warning/risk/blocked/manual/test/calc/system-error`.
- Единые interaction states для кнопок, полей, таблиц, выбранных строк и служебных состояний.
- Общие правила motion и ограничения на анимации в плотных производственных интерфейсах.
- Focus Mode для плотных модулей без изменения данных и бизнес-логики.
- Visual QA с автоматическими маркерами проблем и режимом инспектора элемента.
- Модуль `UI-состояния` в группе `UX-макеты` как живая галерея состояний, таблиц, форм, Gantt-паттернов и empty/loading/error.
- Snapshot QA скрипт `npm run qa:visual`, который проходит ключевые модули на едином viewport `macbook-air-15` и сохраняет скриншоты + отчет.
- Stabilization Pass v1 расширил QA-покрытие на `Заказ-наряды`, `Планирование`, `Мастерскую`, `Диспетчерскую`, `Маршрутную карту`, `Спецификации`, `Номенклатуру`, `Сотрудники`, `Снабжение`, `Карту производства`, `Справочники` и focus-варианты плотных модулей.

## Где проверять в интерфейсе

1. Открыть группу `UX-макеты` в левом меню.
2. Открыть модуль `UI-состояния`.
3. Проверить блоки:
   - `Сигналы системы`;
   - `Interaction States Bible`;
   - `Формы и поля`;
   - `Плотные таблицы MES`;
   - `Gantt Design System`;
   - `Empty / Loading / Error`;
   - `Keyboard UX`;
   - `Design QA Snapshots`;
   - `Visual QA Inspector`.
4. Включить `Visual QA` в верхней панели.
5. Для точечной передачи ошибки нажать кнопку с прицелом в панели Visual QA, затем кликнуть проблемный элемент.
6. Нажать `Скопировать отчет для Codex` или раскрыть `Текст отчета для ручного копирования`.

Инспектор копирует модуль, viewport, selector, координаты, размеры, scroll-контейнер, CSS-сводку и автоматические флаги. Это не заменяет аннотацию глазами, но убирает ручной поиск элемента в коде.

## 1. Система статусов и сигналов

Один смысл должен выглядеть одинаково в любом модуле.

После Contract Migration v1 визуальный статус нельзя выбирать локально внутри компонента. Правило:

- статус объекта читается как `scope + value`;
- подпись, тон и сигнал берутся через `getMesStatusView(scope, value)`;
- выпадающие списки статусов строятся через `getMesStatusOptions(scope)`;
- новые переходы между модулями сначала описываются в `MES_FLOW_TRANSITIONS`.

Это защищает систему от ситуации, когда одинаковое слово, например `partial`, означает разное в заказ-наряде и факте диспетчерской, но случайно рисуется одинаково или влияет не на тот модуль.

| Сигнал | Когда использовать | Визуальный смысл |
| --- | --- | --- |
| `ready` | готово, выполнено, доступно к следующему действию | зеленый спокойный сигнал |
| `warning` | ожидание, частичная готовность, нет критического сбоя | желтый предупреждающий сигнал |
| `risk` | возможный срыв срока, дефицит, конфликт планирования | оранжевый риск |
| `blocked` | действие невозможно, процесс остановлен | красный блокирующий сигнал |
| `manual` | требуется ручное вмешательство диспетчера или технолога | фиолетовый управляемый сигнал |
| `test` | UX-макет, экспериментальная функция | желтый тестовый сигнал |
| `demo-function` | поле или действие можно трогать в прототипе, но оно не влияет на расчеты | темная плашка с белым текстом |
| `calc` | расчетное поле, расчетная база операции, производительность | голубой расчетный сигнал |
| `system-error` | ошибка синхронизации, сохранения или выполнения | красный системный сигнал |

## 2. Interaction States Bible

| Состояние | Правило |
| --- | --- |
| `normal` | нейтральная поверхность, без декоративного шума |
| `hover` | легкое изменение фона или границы, без скачка размера |
| `active` | короткое нажатие, допустим `translateY(1px)` только для кнопок |
| `focus` | единый синий focus ring через `:focus-visible` |
| `disabled` | opacity около 0.5, cursor `not-allowed`, без hover-эффектов |
| `loading` | мягкий sweep/shimmer, не на плотных строках таблиц без нужды |
| `dirty` | локальная подсветка измененной формы или строки |
| `saved` | краткий toast/status, без изменения структуры экрана |
| `error` | красный сигнал + текст ошибки рядом с проблемным полем |
| `warning` | желтый сигнал, если действие возможно, но требует внимания |
| `selected` | фон выбранности + видимая рамка, особенно в таблицах и Ганте |

## 3. Motion Guidelines MES

| Сценарий | Длительность | Правило |
| --- | ---: | --- |
| hover/focus | 120-160 ms | только фон, цвет, border, shadow |
| раскрытие дерева | 180-220 ms | высота/opacity, без горизонтального сдвига |
| dropdown/modal | 160-220 ms | появление без изменения ширины родителя |
| сохранение строки | до 260 ms | мягкая подсветка или toast |
| выбор слота Ганта | 120-180 ms | ring/outline, без сдвига колбаски |
| обновление данных | до 300 ms | локальный loading, не блокировать весь экран |

Где не анимировать: ширина таблиц, ширина колонок Ганта, scroll-контейнеры, критические кнопки управления, resize/drag во время перетаскивания. `prefers-reduced-motion` должен сводить анимации к минимуму.

## 5. Focus Mode

Режим фокуса скрывает вторичные панели и оставляет рабочую область. Он нужен для плотных экранов:

- Планирование с диаграммой Ганта;
- маршрутная карта;
- спецификации и состав изделия;
- заказ-наряды;
- мастерская и диспетчерская;
- снабжение;
- справочники и таблицы при массовом просмотре.

В v1 режим не меняет данные и не отключает действия. Он управляет только layout-видимостью. В Планировании с диаграммой Ганта доступен отдельный компактный tray, потому что стандартный topbar там скрыт ради рабочей области.

## 6. Visual Debug Overlay

QA-режим подсвечивает подозрительные места прямо в интерфейсе:

- глобальный горизонтальный overflow страницы;
- элементы, выходящие за viewport по X;
- dropdown/modal/popover вне экрана;
- слишком маленькие touch-target;
- текст, который может быть обрезан контейнером.

Отчет доступен в панели и через копирование. Это ускоряет передачу дефектов в Codex: вместо длинного описания можно получить селектор, координаты, размер и тип проблемы.

Режимы:

- автоматический список проблем - показывает overflow, маленькие target-зоны и floating-слои вне viewport;
- инспектор элемента - позволяет выбрать конкретный проблемный элемент и скопировать отчет именно по нему;
- fallback textarea - если браузер не дал доступ к clipboard, отчет остается в поле для ручного копирования.

## 7. Empty / Loading / Error States

Пустые состояния должны быть одинаковыми по тону:

- пунктирная рамка;
- спокойный нейтральный фон;
- короткий текст;
- без больших декоративных карточек;
- без скрытия критических действий.

Loading должен быть локальным. Error должен быть рядом с проблемным объектом и не должен выглядеть как обычный warning.

## 8. Компонентные паттерны для плотных таблиц

Правила для MES-таблиц:

- таблица может иметь внутренний горизонтальный scroll, вся страница - нет;
- actions-колонка sticky справа, если строка шире рабочей области;
- compact chips вместо длинного текста в узких колонках;
- tree-cell для иерархий изделия и маршрута;
- inline edit не должен менять высоту строки скачком;
- dropdown должен выходить во viewport-popover, если не помещается внутри таблицы;
- строки должны быть плотными, но touch target на мобильном не меньше 36 px.
- demo-function поля в таблицах выделяются отдельным demo-стилем и не должны выглядеть как реальные расчетные данные.
- рабочие расчетные поля, например трудозатраты заказ-наряда, не окрашиваются как demo и должны проверяться функциональными QA-сценариями.

## 9. Gantt Design System

Правила для колбасок:

- скругление означает реальное начало или реальное окончание операции;
- разрыв нерабочим временем остается с прямым углом;
- non-working segment показывает перенос операции, а не завершение;
- выбранный слот получает ring/outline, геометрия не сдвигается;
- частичная передача объема отображается отдельным индикатором под основной колбаской;
- риск/конфликт должен менять тон колбаски, но не ломать читаемость текста;
- зависимости должны читаться слабее слотов, но оставаться видимыми при выделении.

## Проверка перед merge/push

Минимальный чеклист:

1. Layout viewport `1710x1112` (`macbook-air-15`): регулярная проверка верстки идет только на этом эталонном экране.
2. Mobile/tablet viewport запускать только по отдельной задаче на адаптивность, не как часть обычного layout gate.
3. Включить Focus Mode на модуле, который менялся, если в нем есть сложная рабочая область.
4. Включить Visual QA и просмотреть критические замечания на измененном модуле.
5. Запустить `node --check src/app.js`, `git diff --check`, `node scripts/build.mjs`.
6. Запустить `npm run qa:visual`, если правились системные UI-паттерны или `UI-состояния`.
7. Запустить `node scripts/scroll-dropdown-qa.mjs`, если правились таблицы, формы, dropdown или scroll-контейнеры.

## Snapshot QA

Команда:

```bash
npm run qa:visual
```

Что проверяется:

- `macbook-air-15` - `1710x1112`.

Это единственный регулярный viewport для layout QA. Проверки mobile/tablet остаются отдельными сценариями и запускаются только когда задача явно касается адаптивности.

Ключевые модули:

- `UI-состояния`;
- `Планирование`;
- `Заказ-наряды`;
- `Мастерская`;
- `Диспетчерская`;
- `Маршрутная карта`;
- `Спецификации`;
- `Номенклатура`;
- `Сотрудники`;
- `Снабжение`;
- `Карта производства`;
- `Справочники`.

Скрипт сохраняет отчет и скриншоты в `tmp/design-qa-snapshots-<timestamp>/`. Ошибкой считаются глобальный горизонтальный overflow, элементы вне viewport, маленькие touch target, floating/dropdown вне экрана и явное текстовое переполнение.

Помимо базового открытия модулей, в `qa:visual` нужно добавлять отдельные интерактивные состояния для блоков, которые уже ломались или критичны для работы. Текущий пример: `routes-labor-open` раскрывает трудоемкость операции в маршрутной карте и проверяет этот блок как самостоятельное состояние.

Внутренний горизонтальный scroll допустим только для плотных рабочих зон: таблицы, временная шкала Планирования/Ганта, снабженческий Gantt, производственная flow-лента карты цеха. Вся страница при этом не должна получать горизонтальный scroll.

Для диаграммы сотрудников desktop сохраняет горизонтальные уровни и стрелки. На мобильных ширинах допускается wrapped-раскладка слоями без стрелок, чтобы текст оставался читаемым и не превращался в вертикальные полосы.
