# MES Stabilization Pass — 2026-06-29

## Цель

Закрепить текущий прототип как связанную MES-систему перед следующим расширением бизнес-логики: производственная структура идет через `Модуль сотрудники`, трудозатраты заказ-наряда ведут расчет плановых слотов, `Табель` ограничивает доступность исполнителей, `Мастерская · доска` распределяет сменные задачи и передает визуальный operational-layer в Gantt.

## Зафиксированные контракты

### Производственная структура

- `src/production_structure_matrix_data.js` — источник строк производственной матрицы.
- `src/production_structure_service.js` — сервис преобразования матрицы в рабочие центры, ресурсы, сотрудников, мастеров и исполнителей.
- Старые производственные справочники `departments`, `resources`, `equipment`, `productionResources`, `norms`, `employees` не должны возвращаться в видимые справочники.
- Мастера получают область распределения через матрицу мастеров: ветка, точный участок, ручной список или все исполнители.
- `Мастерская · доска` должна сохранять assignment только по исполнителям, которые попали в разрешенную область матрицы выбранного мастера. Ручной режим матрицы сначала расширяет/сужает разрешенный список, а прямой submit чужого employeeId отбрасывается.
- Перед записью распределения выбранный мастер и список разрешенных исполнителей пересчитываются заново через матрицу; UI-строка доски не считается достаточным источником доверия.
- Сотрудники верхнего административного уровня без производственного участка нормализуются в виртуальный `Административный отдел`, чтобы авторизация и сайдбар показывали ФИО, должность и отдел без дублирования должности.

### Планирование

- Будущие/незакрытые слоты Gantt должны брать длительность из трудозатрат заказ-наряда (`planningLaborSource: "work_order"`).
- Старые расчетные поля `unitsPerHour`/`capacity` остаются только как compatibility/fallback и для специальных расчетов, но не должны быть основным источником активных плановых слотов.
- Изменение трудозатрат заказ-наряда должно сохранять scroll и пересчитывать будущий слот через helper-слой.

### Оперативный контур

- `Табель` хранит доступность сотрудников, графики и исключения.
- `Мастерская · доска` распределяет сменные задачи по доступным исполнителям и формирует слой `план / распределено / факт / остаток`.
- Выбор исполнителя в `Мастерской · доска` одновременно ограничен матрицей мастера и доступностью из `Табеля`: матрица отвечает на вопрос "кого мастер имеет право назначать", табель — "кто доступен в эту смену".
- Новые назначения, факты и остатки Мастерской пишутся только в `ui.shiftMasterBoardAssignments`, `ui.shiftMasterBoardFacts`, `ui.shiftMasterBoardCarryovers`. Старый `planningState.shiftMasterAssignments` остается только compatibility/read-слоем для старых сохранений и Gantt-совместимости.
- `Диспетчерская` сейчас является заглушкой и не участвует в рабочем контуре.
- Факт смены логически считается архивом факта, а не активной Диспетчерской.

### Авторизация

- `authPrototype` — системный вход, не обычный пункт UX-макетов в главном меню.
- Неправильный PIN не создает unlocked-сессию.
- PIN `55555` открывает стандартный shell и сохраняет сессию до конца дня.
- Тестовый PIN не должен отображаться в интерфейсе.
- PIN-клавиатура должна показывать 10 уникальных цифр в случайном порядке, без кнопки `C/С`; удаление — отдельная backspace-кнопка.
- Роль интерфейса после входа берется из авторизованного сотрудника, а нижний блок сайдбара обязан показывать ФИО, должность, отдел и кнопку выхода.
- Выход очищает unlocked-сессию и возвращает на первый шаг авторизации.
- `workflow-preset.json` не должен хранить transient-состояние авторизации: выбранного сотрудника, `authPrototype*`, `authGate*`, `authCurrent*` или роль `operator`.
- Настройки модуля `Роли` (`accessRoleProfiles`, `accessRoleAssignments`) входят в `shared-state.sharedUi`, потому что они уже влияют на видимость модулей после входа.

## UI и CSS

- `styles.css` является только manifest-файлом с фиксированным порядком `@import`.
- Модульные CSS-слои живут в `styles/layers/*`.
- Общие UI-kit правила живут в `styles/mes-ui-core.css`.
- Runtime маркирует базовые компоненты через `data-ui-component`.
- Табличные wrapper-ы должны иметь `data-scroll-contract="horizontal-only"` и не владеть внутренним вертикальным scroll.
- Возврат старого поиска, breadcrumbs, update-popup, старых Мастерских, РКД, Склад-модуля, old dashboard/reports/debug и standalone shell запрещен QA.
- `workflow-preset.json` также входит в legacy/UI guard: он не должен возвращать старые UI-классы, старые auth-ключи или удаленные module aliases.

## QA Gates

- `qa:syntax` — синтаксис runtime, contracts, services и QA-скриптов.
- `qa:architecture` — flow, UI, legacy, CSS, production structure.
- `qa:functional` — state, shared-state, module-smoke, planning-labor, shift-master-board, timesheet, Gantt operational layer, shift-flow, auth, roles, boot.
- `qa:nonvisual` — полный non-visual gate для автономных проходов.
- `qa:visual` — MacBook Air 15, один эталонный viewport.
- `qa:visual` дополнительно проверяет реальный экран авторизации без `qa-auth-bypass`: отдел, участок, сотрудник, PIN.
- `qa:visual` включает системные модули `Модуль сотрудники` и `Роли`, а также раскрытый ручной режим матрицы распределения мастеров.
- `qa:css` — CSS graph, manifest-only root stylesheet, duplicate budgets, broad `!important` budget, legacy selector pressure.
- `qa:auth` — keypad digits/backspace, wrong PIN, correct PIN, daily session reload, auth-bound role card, sidebar auth card and logout.
- `qa:roles` — browser-проверка модуля `Роли`: выбор роли, редактирование паспорта, изменение permission, назначение роли сотруднику и сохранение после reload.
- `qa:shift-flow` — сквозная browser-проверка оперативного контура: реальный слот планирования открывается в Мастерской, распределяется по Табелю, сохраняет дефицит, закрывает факт, создает остаток смены, получает трудозатраты из заказ-наряда и отображает operational layer в Ганте.

## Functional Fixes Found By Shift Flow

- В `Мастерская · доска` поле `Ответственный` теперь сохраняет id владельца конкретной строки/участка через `data-shift-board-master-field`. Раньше readonly-поле не передавало masterId, и сохранение могло брать последний активный `ui.activeShiftMasterId`, из-за чего сотрудники корректного участка отфильтровывались матрицей доступа.

## Остаточный долг под контролем

- `projectId` и `batchId` остаются как compatibility aliases и отслеживаются budget-ами.
- CSS graph еще содержит same-context duplicate selector groups; exact duplicate rules равны нулю.
- `material-transfer-slot` остается live-маркером передаточных/складских операций Gantt, не модулем `Склад`.
- `shiftMaster`, `shiftMasterContext`, `shiftMasterV2` остаются только URL aliases на `shiftMasterBoard`.
- Генерация номера сменного заказ-наряда `СЗН-*` продублирована в сборке строк Мастерской: реальные строки из слотов Ганта используют хвост `slot.id`, fallback-строки используют порядковый `index + 1`. Нужно вынести единый helper генерации номера СЗН и переиспользовать его для строк, sheetContract и будущей печати.

## Рабочее правило для следующих задач

Новые модули и крупные UI-блоки добавлять только через текущий shell/UI-kit/runtime-contract. Если нужен новый паттерн, сначала добавить helper/CSS-contract/QA-gate, потом использовать его в модуле.
