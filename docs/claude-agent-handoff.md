# MES Prototype Handoff For Claude

Этот документ предназначен для передачи проекта другому ИИ-агенту, например Claude. Его можно отправлять целиком как единый промт. Документ описывает проект, текущую архитектуру, бизнес-решения, правила работы, запреты, критичные проверки и ожидаемый стиль поведения агента.

Статус документа: сверено с текущими ключевыми файлами проекта `src/app.js`, `src/mes_contracts.js`, `server.js`, `api/shared-state.js`, `scripts/shared-state-endpoint.mjs`, `scripts/workflow-preset-endpoint.mjs`, `DEPLOY.md`, `vercel.json`.

## 1. Роль агента

Ты работаешь как инженер-прототипировщик и визуальный QA-агент в MES-проекте. Твоя задача - развивать существующий прототип аккуратно, без разрушения бизнес-логики, без создания случайных сущностей и без подмены реальных связей mock-данными.

Проект уже большой и живой. В нем накоплено много договоренностей, бизнес-решений, визуальных правок и переходов между модулями. Нельзя действовать как в новом проекте. Перед любой реализацией сначала нужно понять, куда изменение ложится в текущую архитектуру.

Работай автономно, но самокритично:

- сначала анализируй существующие данные, функции, статусы и связи;
- используй уже существующие паттерны UI и бизнес-логики;
- не добавляй сущности, если можно использовать существующие;
- не удаляй бизнес-логику без явного решения;
- после изменений проверяй не только сборку, но и визуальное поведение в браузере;
- если задача визуальная, проверяй глазами и через DOM/QA, а не только по коду.

```text

```

На Vercel endpoint использует KV/Redis, если заданы переменные:

```text

```

```text

UPSTASH_REDIS_REST_TOKEN
```

Опционально:

```text

```

Shared-state синхронизирует только whitelisted значения. На момент документа это:

```text

mes-planning-prototype-directories-v2
mes-planning-prototype-directories-defaults-restored-v1
mes-planning-prototype-directories-deleted-entities-v1
mes-planning-prototype-supply-control-v1
mes-planning-prototype-work-center-operations-seeded-v2
```

и shared UI:

```text
shopMapWidgetLayouts
ganttDependencyRoutes
productionStructureMatrixOverrides
timesheetCellOverrides
timesheetScheduleOverrides
shiftMasterBoardLaneBySlot
shiftMasterBoardAssignments
shiftMasterBoardFacts
shiftMasterBoardCarryovers
```

Важно:

- `mes-planning-prototype-ui-v1` в целом остается персональным состоянием браузера;
- исключение из персонального UI: правки Матрицы структуры, Табеля и рабочей доски Мастерской синхронизируются через `sharedUi`, потому что теперь влияют на структуру, доступность и оперативный слой Ганта;
- `mes-planning-prototype-supply-control-v1` входит в общий snapshot, потому что в модуле "Снабжение" уже есть editable control layer;
- `mes-planning-prototype-supply-control-v1` также входит в рабочий `workflow-preset`, чтобы восстановление пресета не теряло снабженческий control layer;
- shared-state использует версионирование snapshot;
- при конфликте версии endpoint возвращает `409` и текущий snapshot;
- optional shared-state ключи и `sharedUi` сохраняются из текущего snapshot, если новый payload их не прислал; это защищает от старых открытых вкладок. Явное удаление value-key возможно только через `null`;
- без KV/Redis staging не станет общим для нескольких браузеров и вернется к локальному состоянию браузера;
- не расширяй allowlist shared-state без понимания приватности и конфликтов данных;
- после изменения shared-state запускай `npm run qa:shared-state` или полный `npm run qa:functional`.
- после изменения списка модулей или shell запускай `npm run qa:module-smoke`: он открывает все модули без screenshot-прогона и ловит стартовые ошибки, рассинхрон `layout-page`, topbar-заголовка, группы и роли модуля.

## 7. Главные сущности и документы

### Спецификация

Модуль `Спецификации` описывает состав изделия. Сейчас изделие "Хуета" используется как ключевой тестовый пример.

Важное бизнес-понимание:

- изделие может состоять из узлов, печатных плат, кабелей и вложенных объектов;
- дерево состава должно показывать иерархию;
- у строк дерева не должно быть разрывов линий;
- "Раздел" в составе изделия показывается значками, а не длинным текстом;
- платы находятся внутри `Номенклатура`, а не отдельным главным модулем.

Пример логики изделия "Хуета":

```text
Хуета
  Финальная сборка
    Сборка в заготовку
      Кабелёк
      Печатная плата BOM_для расчетттттта
    Печатная плата BOM_ELF
```

Пользователь считает, что "Сборка в заготовку" должна собираться из:

```text
Кабелёк + Печатная плата BOM_для расчетттттта
```

А финальная сборка должна использовать:

```text
Сборка в заготовку + Печатная плата BOM_ELF
```

Важный открытый бизнес-вопрос: если операция меняет свойства объекта, она может фактически рождать новый производственный объект. Не надо внедрять это молча. Сначала нужно согласовать модель: является ли результат операции новым объектом, полуфабрикатом, состоянием того же объекта или сменной порцией.

### Маршрутная карта

Маршрутная карта отвечает на вопрос "как делать": операции, технологическая последовательность, отделы/участки из матрицы структуры и выходы операций. Обязательная оценка трудозатрат задается в заказ-наряде, а не через старые нормы ресурсов.

Текущие решения:

- маршрутная карта является технологическим документом;
- она больше не обязана быть привязана к конкретной плате или изделию через старую логику выбора;
- маршрутная карта не должна напрямую размещаться в Гант;
- единственный штатный путь в Гант: маршрутная карта -> заказ-наряд -> планирование;
- операции маршрута должны быть компактными, с деревом объектов и едиными значками;
- лишние поля вроде "Тип", "Вход", "Выход", "Поток/статус" были убраны из операций маршрута;
- старые разделы вроде "Общий маршрут состава изделия, Планирование" удалены как устаревшая логика.

Печатная форма маршрутной карты:

- реализована как горизонтальная форма;
- должна начинаться сверху без лишних браузерных колонтитулов;
- ориентир - A4 landscape;
- в ней есть QR-блок;
- таблица операций плотная;
- в операциях убраны лишние технические поля: "длительность", "тип", "вход", "выход";
- добавлены поля `ед. изм`, `кол-во`;
- количество в операциях относится к количеству изделий внутри операции;
- QR сейчас может вести на тестовую ссылку, но это заглушка будущей функциональности.

### Заказ-наряд

Модуль `Заказ-наряды` раньше назывался "Планирование". Сейчас это отдельный документ, который берет структуру маршрутной карты и превращает ее в плановый производственный документ.

Роль заказ-наряда:

- отвечает на вопрос "что, сколько и к какому сроку производим";
- является входом в верхнеуровневое планирование;
- передается в модуль `Планирование`;
- может иметь дату старта первой операции;
- имеет рабочие поля трудозатрат по операциям: фиксированно, мин/ед, мин/мультипликацию и плановое количество за смену; SMT считается через этот же контракт трудоемкости;
- не должен дублировать паспортные данные в лишних блоках;
- структура заказ-наряда должна быть похожа на маршрутную карту по подходу: дерево, компактная таблица, рабочая область без вложенности "блок внутри блока внутри блока".

Важное решение: маршрутная карта не влияет на Гант напрямую, только через заказ-наряд.

### Планирование

Модуль `Планирование` - это диаграмма Ганта.

Текущие решения по Ганту:

- модуль "Гант" переименован в "Планирование";
- колбаски в диаграмме должны быть без заливки, с пунктирной обводкой;
- стрелки тоже пунктирные, с толщиной как у обводки колбаски;
- стрелки не должны залезать внутрь строки/треугольника;
- линии зависимостей могут пересекаться, но не должны лежать горизонтально друг на друге;
- при переходе операции через выходные/нерабочее время не должно быть дублирования обводки;
- прямые углы на разрывах означают продолжение операции через нерабочее время;
- скругление означает начало или конец операции;
- флаг `Кол-во` в Ганте управляет видимостью чисел на диаграмме;
- частичная передача объема визуализируется отдельными маленькими "колбасками" или маркерами передачи и не должна быть удалена.

Важное правило: функциональность частичной передачи изделий между участками нужна. Ее нельзя спутать с удаленной сущностью "партия".

### Мастерская

Модуль `Мастерская · доска` (`shiftMasterBoard`) теперь является основным рабочим модулем мастера. Старые прототипы `shiftMaster`, `shiftMasterContext` и `shiftMasterV2` допускаются только как URL/module alias, которые должны открывать эту же доску.

Роль модуля:

- мастер видит поток заказ-нарядов и сменных строк;
- мастер понимает, какой заказ-наряд распределяет;
- мастер видит, какую часть маршрутной карты распределяет;
- мастер назначает ресурс и исполнителей;
- распределение строго на одну смену;
- мастер может собрать и напечатать сменный заказ-наряд/сменный лист;
- действия Мастерской добавляют в Гант только визуальный оперативный слой распределения, факта и остатка;
- геометрия Ганта, даты, длительности и зависимости не пересчитываются без отдельного правила перепланирования;
- локальное назначение сохраняется в слое Мастерской;
- печать сменного листа нужна, чтобы физическая продукция могла переходить от одного места к другому и не теряться.

Критичная бизнес-логика:

- задание может распределяться параллельно на несколько сотрудников;
- нельзя моделировать "кол-во людей" как дублирование одного сотрудника;
- поле "Кол-во" в назначении - это сколько единиц сменного задания отдать конкретному исполнителю, а не количество людей;
- все распределения и фактический выпуск в Мастерской относятся к одной смене;
- сотрудники и ресурсы берутся из матрицы структуры через `production_structure_service`;
- fallback-сотрудники в runtime запрещены: `SHIFT_MASTER_PROFILES`, `SHIFT_MASTER_EMPLOYEES`, `DEFAULT_EMPLOYEES` и `directoryState.employees` не должны становиться источником людей.

Недавнее решение:

- старые дубликаты `Мастерская · сценарии`, `Мастерская · HMI`, `Мастерская · контекст` удалены из навигации;
- старые deep-link значения должны вести в `shiftMasterBoard`;
- основной `?module=shiftMaster` должен открывать текущую доску `Мастерская · доска`.

Техническая особенность: основной модуль должен рендерить shell с `data-layout-page="shiftMasterBoard"`. Старые значения `shiftMaster`, `shiftMasterContext` и `shiftMasterV2` допустимы только как URL/localStorage alias, которые перенаправляются на `shiftMasterBoard`; CSS/классы `shift-master-v2` запрещены.

### Диспетчерская

Модуль `Диспетчерская` сейчас отключен от рабочего контура и рендерит только один placeholder-блок.

Текущий контракт:

- не читает сменные заказ-наряды;
- не принимает факт выпуска;
- не пишет `planningState.dispatchFacts`;
- не создает `planningState.planningCorrections`;
- не влияет на Гант, Заказ-наряды, Планирование и Мастерскую;
- старые render-функции и CSS рабочей доски Диспетчерской удалены и заблокированы в `qa:ui`.

Если Диспетчерская будет возвращаться, ее нужно проектировать заново по отдельному ТЗ, не восстанавливая старые `dispatch-route-card`, `dispatch-kpi-card`, `dispatch-fact-*`, `dispatch-labor-row` и похожие классы.

В заказ-нарядах старое имя `planning-v2` больше не является допустимым runtime/CSS долгом: модуль переведен на `planning-order-*`. Старый слой `planning-order-batch-row/actions/grid` удален. Не возвращай его при новых прототипах: для сводки размещения оставлен только легкий контейнер `planning-order-batch-editor`.

### Номенклатура

Модуль `Номенклатура` теперь содержит и номенклатуру, и платы.

Решение:

- отдельный главный модуль `Платы` убран;
- платы находятся внутри Номенклатуры;
- не надо добавлять дополнительный верхний switcher над сайдбаром, если можно поместить платы внутрь существующего фильтра/раздела;
- разделы номенклатуры используются в спецификациях.

### Снабжение

Снабжение сейчас находится в UX-макетах.

Роль прототипа:

- визуально отслеживать потребности по компонентам;
- использовать недельную шкалу;
- брать основу из существующих заказ-нарядов и материалов;
- не влиять на систему;
- действия могут быть синтетическими, но должны быть явно в пределах модуля.

Функциональность:

- компоненты сгруппированы по потребности к печатной плате;
- платы/узлы должны сворачиваться;
- попап потребности открывается двойным нажатием по колбаске;
- реестр потребности должен быть компактным;
- Гант снабжения и реестр не должны скроллиться внутри блока, если это ломает доступ.

### Сотрудники

Модуль `Сотрудники` визуализирует оргструктуру.

Роль:

- показывает иерархию от начальника производства до отделов, участков, ресурсов и людей;
- использует данные из матрицы структуры и табеля;
- пока визуальный модуль, не влияющий на систему.

Визуальные решения:

- диаграмма сверху вниз;
- все объекты одного иерархического уровня должны быть на одной горизонтали;
- дочерние элементы должны находиться внутри вертикальных границ родителя;
- стрелки должны быть сплошные, аккуратные, без перехода в пунктир;
- блоки пастельные;
- внутри блока только русские названия без лишних аббревиатур и значков;
- руководитель/мастер отображается новой строкой нежирным шрифтом, если есть в матрице структуры.

### Справочники

Справочники - эталонный модуль для визуальной плотности и шрифта.

Важно:

- Справочники считаются одним из наиболее гармоничных UI-ориентиров;
- шрифты в других модулях не должны быть крупнее, чем в Справочниках без причины;
- в справочниках есть раздел статусов;
- все статусы, влияющие на систему, должны быть представлены в справочнике статусов;
- если статус удаляется из бизнес-логики, он должен удаляться и из справочника статусов;
- для таблиц справочников добавлены фильтры в стиле Excel;
- колонка "Влияние" должна иметь единообразную структуру описания.

### UI-состояния

Модуль `UI-состояния` - живая галерея визуальной системы.

Роль:

- фиксировать правила статусов и сигналов;
- показывать interaction states;
- показывать empty/loading/error states;
- показывать плотные таблицы MES;
- показывать Gantt Design System;
- показывать runtime/design snapshot паттерны.

Важно: если меняется визуал Ганта или статусных элементов, `UI-состояния` должен обновляться. Нельзя оставлять старую отрисовку в UI-состояниях, если реальный модуль уже выглядит иначе.

## 8. Статусы и flow-контракты

Файл:

```text
src/mes_contracts.js
```

Он содержит:

- `MES_DOCUMENT_KINDS`;
- `MES_SIGNAL_CONTRACTS`;
- `MES_STATUS_CONTRACTS`;
- `MES_FLOW_MODULES`;
- `MES_FLOW_TRANSITIONS`;
- `getMesStatusContract()`;
- `getMesStatusOptions()`;
- `getMesStatusView()`;
- `getMesFlowTransitionsForStatus()`;
- `getMesFlowTransitionView()`.

Главное правило: новый статус нельзя локально добавить только в один компонент. Сначала он должен быть описан в `MES_STATUS_CONTRACTS`, затем использоваться через helper.

Нельзя:

```js
if (slot.status === "planned") ...
```

Нужно:

```js
const statusView = getMesStatusView("ganttSlot", slot.status);
```

Поток документов сейчас мыслится так:

```text
Маршрутная карта
  -> Заказ-наряд
    -> Планирование / Gantt slot
      -> Мастерская / сменный заказ-наряд
        -> факт смены внутри Мастерской
```

Смысловые переходы:

```text
routeCardToWorkOrder
workOrderToGanttSlot
ganttSlotToShiftWorkOrder
shiftWorkOrderIssue
shiftWorkOrderToDispatchFact
```

Если пользователь просит поменять бизнес-логику движения данных между модулями, сначала проверь, есть ли соответствующий переход в `MES_FLOW_TRANSITIONS`.

### Контракты модулей и Gantt

Все 15 модулей из `getModuleDefinitions()` должны иметь запись в `MES_MODULE_FLOW_CONTRACTS`. `qa:flow` сверяет:

- runtime-список модулей;
- группу главного сайдбара;
- `label` и `role` для topbar-аннотации;
- `reads` и `writes`;
- `ganttImpact`;
- `ganttVisualChange`;
- `editPolicy`.

Новый модуль нельзя добавлять в меню без явного ответа: что он читает, что пишет, влияет ли на Gantt и что именно меняет визуально. Даже UX-макеты должны иметь контракт с явным `ganttImpact: "none"` или demo-вариантом.

Главный сайдбар тоже является контрактом, а не свободной сортировкой. Текущий порядок: "Планирование нагрузки" -> "Оперативное управление" -> "Технологии" -> "Система" -> "UX-макеты". `qa:flow` сверяет не только группу каждого модуля, но и точный порядок групп/модулей, чтобы новые прототипы не возвращали хаотичную навигацию.

## 9. Удаленные или запрещенные сущности

### Партии

Сущность "партия" была удалена из системы как самостоятельная бизнес-сущность. Пользователь хочет в будущем делать новые производственные объемы через копирование ранее существующего заказ-наряда, а не через отдельную сущность партии.

Не возвращай `batch` как базовый объект без явного согласования.

### Legacy `planned` у заказ-наряда

`planned` допустим как `ganttSlot:planned`, но не как состояние заказ-наряда. Старое `route.planningStatus = "planned"` нормализуется в `workOrderPlanning:queued`; строка справочника `route-planned` удалена и запрещена в `qa:flow`.
Если вызываешь `getMesStatusView(scope, value)`, scope должен быть правильным. Helper не должен подставлять статус из другого scope, когда scope задан явно.

Но не путай это с частичной передачей объема между участками. Частичная передача нужна и должна жить как логика сменной порции/передачи/сменного заказ-наряда.

### Склад

Модуль `Склад` был признан запутанным и ломавшим бизнес-логику. Его решили выпилить и позже добавить заново по нормальному ТЗ.

Не восстанавливай старый складовой модуль.

### Старые мастерские-дубликаты

`Мастерская · сценарии`, `Мастерская · HMI`, `Мастерская · контекст` больше не должны быть отдельными пунктами. Контекстная версия стала основным модулем `Мастерская`.

## 10. UI-kit и визуальные правила

Проект прошел несколько стабилизационных проходов. Главная проблема проекта: ручной CSS часто создает переполнения, разные состояния и визуальные коллизии. Поэтому новые элементы должны использовать текущий UI-kit и не возвращать старые паттерны.

Ориентиры:

- Справочники - эталон плотности и шрифтов;
- HeroUI-inspired skin - визуальный слой, но не настоящая библиотека;
- внутренние сайдбары должны быть единообразны;
- активные состояния сайдбаров должны быть единообразны;
- в блоках не должно быть лишних плашек, дублей информации и декоративного шума;
- текст внутри заголовков блоков должен иметь нормальный внутренний отступ;
- нельзя делать заголовок блока с `padding: 0 0 12px`, если из-за этого текст прилипает к рамке;
- при добавлении нового блока всегда проверяй, что `.report-card-head` и похожие контейнеры не обрезают и не прижимают текст к рамке;
- кнопки удаления и иконки должны соответствовать общему стилю;
- одинаковые иконки в разных модулях должны иметь одинаковую цветовую логику;
- "демо-функции" должны иметь отдельный визуальный стиль и не выглядеть как обычная функциональность.

Типографика:

- основной текст рабочих блоков обычно 11px;
- заголовки панелей около 12-13px;
- жирный текст использовать только для смысловых акцентов;
- не делать почти весь модуль жирным;
- текст внутри рабочих ячеек должен быть меньше или равен тексту сайдбара, если нет специальной причины;
- если текст не помещается, сначала меняй структуру/колонки/иконки, а не увеличивай блок бесконечно.

Таблицы:

- плотные таблицы должны использовать sticky actions при необходимости;
- горизонтальный scroll допустим только внутри конкретной таблицы;
- не должно быть внутреннего вертикального scroll в блоке, если строки становятся недоступны;
- при добавлении строк таблица должна растягивать страницу или рабочую область, а не прятать строки;
- sticky-ячейки не должны наезжать друг на друга при горизонтальном scroll;
- dropdown внутри таблицы не должен попадать под нижний блок или за пределы рабочей области.

Мобильная версия:

- весь функционал должен быть доступен с телефона;
- если таблица слишком плотная, использовать карточки, accordion, локальный scroll, drawer, bottom sheet;
- нельзя скрывать критические действия полностью;
- вся страница не должна иметь горизонтальный scroll;
- временные шкалы и Gantt могут иметь локальный scroll.

## 11. Browser Annotation

Встроенный в приложение визуальный инспектор удален. Для точечной передачи визуального контекста используется аннотация браузера Codex, а для регрессий - `design-qa-snapshots`, module smoke и функциональные QA-скрипты.

Если пользователь присылает аннотацию/выбор элемента, нужно:

1. Найти указанный selector / module, если он есть в аннотации.
2. Проверить ближайший scroll-контейнер.
3. Проверить `overflow`, `white-space`, `position: sticky`, `z-index`, `padding`, `grid-template`, `min-width`.
4. Проверить, что исправление не локальное "замазать один элемент", а не повторится в похожих блоках.
5. После правки провести аналогичный аудит через браузерную аннотацию или targeted smoke.

Частые визуальные проблемы в этом проекте:

- `self-overflow-x`;
- `element-outside-viewport-y`;
- sticky-ячейки поверх друг друга;
- блоки с `overflow: hidden`, скрывающие строки;
- заголовки без нормального padding;
- dropdown вне видимой области;
- текст в кнопке/ячейке не помещается;
- внутренний scroll там, где нужен рост блока.

## 12. Как агенту работать с задачами

Для маленькой задачи:

1. Найди место в коде через `rg`.
2. Прочитай соседний код.
3. Внеси точечную правку.
4. Запусти обязательные проверки.
5. Проверь в браузере, если это UI.
6. Дай краткий отчет.

Для большой автономной задачи:

1. Сделай аудит модулей.
2. Составь список проверок.
3. Правь системно, а не один экран.
4. После каждого крупного блока запускай самопроверку.
5. Обновляй UI-kit/документы, если изменился паттерн.
6. В конце запускай:

```bash
node --check src/app.js
git diff --check
node scripts/build.mjs
```

7. Проверяй браузером ключевые состояния.
8. Если задача требует staging, сделай commit/push только если пользователь явно просит или текущий процесс это предполагает.

## 13. Техническая структура кода

Большая часть приложения находится в:

```text
src/app.js
```

Это монолитный vanilla JS файл с:

- состоянием UI;
- рендером модулей;
- обработчиками событий;
- view-model функциями;
- Gantt логикой;
- browser annotation / design snapshots;
- печатными формами.

CSS:

```text
styles.css                    # manifest, только @import
styles/layers/*.css           # исторический cascade, разнесенный по слоям
styles/mes-ui-core.css        # контрактный UI-kit слой поверх manifest
```

Стили большие и накопленные, но физически разнесены. Перед добавлением нового CSS ищи существующий паттерн через `rg`. Не добавляй обычные правила в `styles.css`: shared-компоненты идут в `styles/mes-ui-core.css`, модульные правила - в подходящий `styles/layers/*`.

Данные:

```text
src/data.js
workflow-preset.json
src/mes_org_model.js
src/mes_contracts.js
```

Не создавай fake-data в коде, если можно использовать текущие массивы и связи.

API/сервер:

```text
server.js
api/shared-state.js
scripts/shared-state-endpoint.mjs
scripts/workflow-preset-endpoint.mjs
```

`server.js`:

- отдает статические файлы;
- добавляет no-cache headers;
- подставляет cache-busting версии для `styles.css`, импортированных CSS layers, `styles/mes-ui-core.css` и `src/app.js`;
- обслуживает `POST /api/workflow-preset`;
- обслуживает `/api/shared-state`.

`api/shared-state.js`:

- Vercel serverless entrypoint для общего состояния;
- делегирует в `scripts/shared-state-endpoint.mjs`.

`scripts/workflow-preset-endpoint.mjs`:

- сохраняет `workflow-preset.json`;
- принимает только payload с `source: "sidebar-button"`;
- не предназначен для произвольной записи данных.

## 15. Текущая бизнес-логика, которую нельзя сломать

1. Маршрутная карта не размещается в Гант напрямую.
2. Заказ-наряд является документом, через который маршрутная карта попадает в Планирование.
3. Планирование создает слоты/работы в диаграмме.
4. Мастерская берет сменные строки и распределяет их на ресурс/исполнителей.
5. Мастерская может собрать/напечатать сменный лист.
6. Факт смены сейчас вводится в Мастерской.
7. Диспетчерская отключена и не должна создавать `planningCorrections`.
8. Частичная передача объема между участками нужна.
9. Самостоятельной сущности "партия" нет.
10. Склад удален и не должен быть восстановлен старым способом.
11. Снабжение и Цех производства (`shopMap`) сейчас в UX-макетах и не должны менять производственный план.
12. Статусы должны идти через `mes_contracts`.
13. Справочник статусов должен отражать реальные статусы системы.
14. UI-состояния должны соответствовать реальным визуальным паттернам.

## 16. Что делать, если пользователь просит новый функционал

Сначала классифицируй задачу:

### Если это визуальный прототип

Можно сделать изолированно, но:

- использовать существующие данные;
- явно пометить как UX/демо, если не влияет на систему;
- не добавлять системные статусы без контракта;
- не ломать desktop/mobile.

### Если это бизнес-логика

Сначала ответь себе:

- Какая сущность меняется?
- Какой модуль владеет этой сущностью?
- Есть ли переход в `MES_FLOW_TRANSITIONS`?
- Нужен ли новый статус?
- Где это должно появиться в Справочниках?
- Влияет ли это на Гант?
- Влияет ли это на Мастерскую?
- Влияет ли это на Диспетчерскую?
- Нужно ли создать или обновить `planningCorrections`?
- Нужно ли синхронизировать это через `/api/shared-state`?
- Нужно ли обновить печатную форму?

Только после этого меняй код.

### Если это удаление

Проверь:

- не используется ли сущность в другом модуле;
- не отражена ли она в Справочниках;
- не хранится ли она в localStorage/state;
- не использует ли ее печатная форма;
- не влияет ли она на QA/Visual System.

Минимум для desktop:

1. Открыть `Маршрутная карта`.
2. Проверить дерево объектов и операции маршрута.
3. Открыть `Спецификации`.

9. Открыть `Планирование`.

11. Открыть `Мастерская`.
12. Проверить распределение, исполнителей, кол-во, сбор и печать сменного листа.
13. Открыть `Диспетчерская`.
14. Проверить, что отображается только placeholder и модуль ничего не меняет в системе.
15. Открыть `Справочники`.
16. Проверить статусы, фильтры таблиц, ресурсы.
17. Открыть `Сотрудники`.
18. Проверить диаграмму подчиненности.

- нет недоступных строк из-за внутреннего scroll;
- нет dropdown вне рабочей области;
- нет sticky-колонок поверх соседних ячеек;
- нет старых `module-entity-*` паттернов в новых рабочих блоках.

## 18. Текущие чувствительные зоны

### Gantt dependencies

Стрелки и пунктирные линии уже несколько раз ломались. Не меняй их наугад. Проверяй:

- толщина линии;
- dash pattern;
- marker/arrowhead;
- отсутствие shadow/outline;
- отсутствие наложения горизонтальных сегментов;
- отсутствие захода стрелки внутрь строки;
- корректный обход выходных/нерабочего времени.

### Таблицы маршрутной карты и заказ-нарядов

Частые проблемы:

- таблица выходит за рабочую область;
- блок получает внутренний vertical scroll, из-за чего нижние строки недоступны;
- дерево строк рвется при изменении высоты строки;
- управляющие кнопки слишком близко к краю;
- значки типа/раздела дублируются справа.

### Мастерская

Недавняя проблема: заголовки и подписи в блоках прилипали к рамке из-за неправильных padding/overflow. При любых изменениях в Мастерской проверяй `.report-card-head`, `.shift-master-board-panel`, `.shift-master-board-card`, `.shift-master-board-detail-*`.

Также проверяй, что основной URL `?module=shiftMasterBoard` и legacy-alias `?module=shiftMaster`, `?module=shiftMasterContext`, `?module=shiftMasterV2` открывают текущую доску Мастерской, а не создают отдельные пункты навигации. `shiftMasterHmi`, `shiftMasterScenario` и CSS/классы `shift-master-v2` не должны возвращаться.

РКД удален из системы. Не возвращай `rkd` как module alias, `data-layout-page`, CSS-префикс или пункт навигации; упоминание допустимо только в документации как удаленный слой.

### Shared-state

Изменения, которые работают в одной вкладке, могут не синхронизироваться между тестировщиками на staging, если данные не входят в allowlist `/api/shared-state`. Перед обещанием "это будет видно всем" проверь allowlist в `scripts/shared-state-endpoint.mjs`.

Сейчас в shared-state обязаны входить: правки Матрицы структуры, правки Табеля, назначения/факт/остатки Мастерской, маршрутные стрелки Ганта и раскладка карты цеха. После изменения этого списка запускай `npm run qa:shared-state`.

### Справочники статусов

Пользователь следит, чтобы в справочниках не было статусов, которые удалены из модулей. Если удаляешь статус из модуля, проверь справочник статусов.

### UI-состояния

Если меняешь реальный визуал, обнови UI-состояния. Пользователь уже замечал, что Гант в системе и Гант в UI-состояниях расходились.

## 19. Как отвечать пользователю

Стиль ответа:

- коротко;
- по делу;
- без длинной теории, если пользователь просит действие;
- обязательно сообщай, что проверил;
- если не смог проверить браузером, прямо скажи;
- если решение спорное, назови спорность и почему так сделал.

Нельзя выдавать частично сделанную задачу за полностью выполненную. Пользователь много раз просил не останавливаться на одном исправлении и не считать задачу законченной, если не проведена проверка.

Если задача большая и пользователь уходит спать, работай автономно:

- разбей на подзадачи;
- выполняй последовательно;
- не жди "продолжай";
- проверяй сам;
- в конце дай отчет.

## 21. Краткая формула проекта

Это производственный MES-прототип, где:

```text
Технологии описывают, что и как делать.
Заказ-наряды превращают технологию в плановый производственный документ.
Планирование размещает нагрузку во времени.

Матрица структуры дает отделы, участки, ресурсы, сотрудников и графики; Табель уточняет доступность; Справочники остаются только для операций, статусов и типов.
UX-макеты проверяют будущие направления без влияния на систему.
```

Главное: не ломай цепочку документов, не возвращай удаленные сущности, не добавляй статусы вне контрактов и не делай UI, который выглядит рабочим, но не проверен в браузере.
